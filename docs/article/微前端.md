# 微前端

## 什么是微前端

微前端（Micro Frontends）架构可以被理解为“前端领域的微服务”。它的核心思想是：将一个庞大、复杂的单体前端应用
拆分为多个独立开发、独立测试、独立部署的小型前端应用，最后通过一个“容器应用”（Main/Host App）将它们集成在一
 起展示给用户。

##### 微前端解决的问题

微前端方案主要是为了解决传统单体前端（Monolithic Frontend）在复杂应用中存在的问题。

1. **技术栈隔离**：

    单体应用一旦选定 React 或 Vue，后续所有模块都必须跟随。微前端允许不同子应用使用不同的技术栈（React, Vue, Angular 甚至原生 JS）。

2. **代码解耦和风险隔离**：
    
    在单体应用中，修改一个角落的全局样式或变量，可能导致另一个模块崩溃。微前端通过**沙箱隔离（Sandbox）**实现了逻辑和样式的物理/逻辑隔离。这种隔离使得
    一个微应用的故障不会影响整个系统，提高了系统的稳定性和可靠性。
    
3. **发布流水线拥堵**：

    单体应用哪怕只改一个错别字，也要全量编译发布，耗时长且风险高。微前端支持独立发布，子应用上线无需通知主应用。

4. **团队协作**：

    对于庞大的单体应用，代码的合并、构建、测试对于团队来讲都是灾难。微前端让团队可以按业务功能横向拆分，每个团队负责自己的子应用。

##### 微前端的主要优势

1. 技术无关性：每个团队可以选择最适合业务场景的工具。
2. 快速增量升级：可以在不重构整个应用的情况下，用新技术开发新功能，或者逐步替换旧模块（如：主应用 React，新功能用 Vue 3）。
3. 独立开发与部署：每个子应用由独立团队负责，按照团队的节奏迭代、部署。 
4. 高复用性：子应用可以像组件一样，被集成到不同的业务系统中。
5. 强隔离性：一个子应用崩溃不会导致整个门户挂掉，容错性更高。

##### 微前端的应用场景

虽然微前端很强大，但它也带来了首屏加载慢（多次资源请求）和管理复杂度增加的问题。因此，它通常适用于以下场景：

1. 超大型项目：由不同的团队开发、共同构建的应用，如企业级的 ERP、CRM、工业软件解决方案等。
3. 老项目重构：使用新技术去开发新功能，而旧项目又不能直接扔掉，可以使用微前端进行“渐进式”迁移。


综上所述：微前端并不是“让应用跑得更快”的技术，它是“让团队协作和代码维
护不随着应用规模扩大而崩盘”的架构方案。如果你的项目只有几个人维护，业务简单，强行上微前端只会增加“架构负担”。


## 微前端框架需要具备的能力



微前端架构中最具技术含量的核心：如何让多个独立运行的应用在同一个浏览器页面里“互不干扰”地和平共处。

在微前端中，隔离主要分为两个战场：样式隔离和JS 沙箱隔离。

##### 样式隔离

如果子应用 A 定义了 .title { color: red; }，而子应用 B 也有一个 .title 且希望它是蓝色的，后加载的样式就会覆盖掉先加载的。

常见的解决方案：

1. Shadow DOM (最硬核的隔离)：

    利用浏览器原生的 Shadow DOM API，将子应用挂载在 Shadow Root 下。这样 CSS 永远不会逃逸出这个容器。
    缺点： 由于绝大多数 UI 组件库会把弹窗挂载到 `body` 下， 而 shadow DOM 内部 CSS 无法影响到外部样式，导致挂载位置出错。

##### JS 隔离

实现 JS 隔离有下面两种方案，但是各自都存在着需要解决的问题。

- `iframe` 实现 JS 隔离。

-  纯手工实现 JS 沙箱。纯手工实现 JS 沙箱需要解决的问题：

    1. 共享上下文环境。主应用和子应用共享上下文环境，访问的是同一个 `window`、`globalThis`。
    2. 全局数据泛滥。JS 框架或者代码里存在大量显性 + 隐性读写全局。
    3. 读写冲突。
       
        - 禁止写：子应用修改全局变量时必须隔离，不能修改真实全局，否则会互相覆盖。
        - 不能禁读：子应用必须能正常使用全局变量，如 `document`、`location`。
   
    4. 全局副作用的清理。

        子应用运行期间在**全局**产生的大量“环境垃圾”需要清理，否则会拖垮主应用。



## 微前端方案：qiankun（乾坤）

[qiankun](https://qiankun.umijs.org/zh) 是基于 single-spa 进行的深度二次封装，解决了 single-spa 过于底层、无沙箱机制、需要手动处理资源加载的痛点。




### 底层原理

qiankun 是基于两个关键库构建的：

1. single-spa：负责路由匹配、应用状态切换。
2. import-html-entry：负责 HTML、JS、CSS 资源的加载、处理。

在此基础上，qiankun 补全了微前端最核心的两个能力：沙箱隔离、应用间通信。


#### JS 沙箱隔离



qiankun 使用 `ProxySandbox` （多例沙箱） 实现子应用 JS 的隔离：即它会为每一个子应用都创建一个 `ProxySandbox` 实例作为沙箱环境。
